# 简历工坊竞品分析与差异化升级方案

> 分析对象：  
> 1. [LiuMengxuan04/shushu-internship-tool](https://github.com/LiuMengxuan04/shushu-internship-tool)  
> 2. [mmmlllnnn/ResumeCollection](https://github.com/mmmlllnnn/ResumeCollection)  
> 3. [Hacker233/resume-design](https://github.com/Hacker233/resume-design)  
>  
> 结合对象：本项目《本地 AI 简历工坊开发落地方案》  
> 分析目标：吸收竞品优点，识别竞品短板，明确我们如何做得更适合“本地接单交付 + 后续小程序化”的路线。

---

## 1. 总体结论

这三个项目分别代表了简历产品的三种典型方向：

| 项目 | 代表方向 | 核心价值 | 对我们的启发 |
|---|---|---|---|
| shushu-internship-tool | AI 求职工作流 | 从 JD 到项目、简历、面试材料的闭环 | 我们不能只做“排版工具”，要做“求职交付工作流” |
| ResumeCollection | 模板资源库 | 大量 Word 简历模板、分类丰富、可直接下载 | 我们需要建立行业模板库和模板标签体系 |
| resume-design / 猫步简历 | 在线简历设计器 | 模板、设计器、主题、导出、后台管理、AI 能力 | 我们需要吸收它的模板/主题/模块化优势，但避免过度复杂 |

我们的正确定位不是正面硬刚“在线简历平台”，而是做一个更细、更快、更适合接单交付的工具：

```text
竞品多是“用户自己做简历”
我们要做“运营/开发者帮客户快速交付好简历”
```

所以我们的差异化不应该只靠模板数量，而应该靠：

1. **本地隐私与接单交付流程**。
2. **AI 辅助 + 人工复核的可控编辑器**。
3. **JD 匹配、简历优化、面试追问的一体化工作流**。
4. **可复用到微信小程序的后端能力和数据结构**。
5. **行业模板 + 内容策略 + 交付 SOP 的组合优势**。

---

## 2. 竞品一：shushu-internship-tool

### 2.1 项目概况

项目说明中明确写到，它的核心定位是“把岗位描述变项目，把项目变简历，把简历变面试”。它面向计算机行业实习候选人，围绕 JD、开源项目选择、项目审计、改造、STAR 简历表达、面试 Q&A 和展示材料形成闭环。

它不是传统简历排版工具，而是一个 AI assistant workflow / skill。README 中列出的能力包括：

- 根据 JD 找 2-3 个 GitHub 项目并排序。
- 审计已 clone 项目，生成 `audit.json`、`overview.md`、`overview.html`。
- 规划本地或远程 baseline run。
- 设计可面试的项目改造点。
- 生成 STAR 简历项目、核心代码讲解、面试官拷问 Q&A、PPT 提示词和投递检查表。

参考来源：[shushu-internship-tool README](https://github.com/LiuMengxuan04/shushu-internship-tool)

### 2.2 优势

#### 优势 1：不是单点工具，而是完整求职闭环

它没有停留在“帮你写一段简历”，而是把链路设计成：

```text
JD 分析 → 项目选择 → 项目理解 → 项目改造 → 简历表达 → 面试准备
```

这对我们非常重要。因为真实付费用户往往不是只想要一份漂亮 PDF，而是想要：

- 简历更匹配岗位。
- 项目经历更像真实做过。
- 面试时能讲清楚。
- 投出去更容易拿面试。

#### 优势 2：有明确的工作流分层

它把流程拆成 intake、repo discovery、candidate ranking、project recon、baseline run、ownership build、interview pack、readiness gate。

这种设计比简单的“一个输入框 + 一个生成按钮”更可靠，因为每一步都有产物、有判断、有检查。

#### 优势 3：强调面试可讲性

项目强调“能投、能面、能讲”，并且要求面试 Q&A 像真实面试官追问。这一点比普通简历生成器强很多。

对于我们后续的增值服务，这可以直接变成高客单价功能：

- 简历优化。
- 项目经历深挖。
- 面试问题预测。
- 面试回答稿生成。
- 项目复盘卡片。

#### 优势 4：评分和排序思路清楚

它不是让 AI 随便推荐，而是把 JD 匹配、license、可运行性、资源适配、用户偏好等拆成显式字段再排序。

这给我们的启发是：我们的“简历评分”和“JD 匹配”也不能只是大模型自由发挥，应该拆成结构化维度。

例如：

| 评分维度 | 分值 | 说明 |
|---|---:|---|
| JD 关键词匹配 | 25 | 是否覆盖岗位核心技能 |
| 经历可信度 | 20 | 是否有明确背景、动作、结果 |
| 量化成果 | 15 | 是否有数字、规模、结果 |
| 结构完整度 | 15 | 教育、工作、项目、技能是否完整 |
| ATS 友好度 | 15 | 是否易解析、少复杂元素 |
| 表达质量 | 10 | 是否简洁、专业、无废话 |

### 2.3 劣势

#### 劣势 1：不是普通用户可直接上手的产品

它更像给 AI agent / Codex 使用的 skill 和脚本工具。普通手机用户、闲鱼客户很难直接理解或使用。

我们的机会：

- 把复杂工作流做成可点击的 UI。
- 用按钮和表单承载流程，而不是要求用户理解脚本。
- 给运营者一个“下一步该做什么”的工作台。

#### 劣势 2：偏计算机实习方向

它对后端、前端、全栈、AI、云原生等计算机方向很强，但对行政、财务、医护、运营、销售、设计等普通岗位覆盖不足。

我们的机会：

- 做通用行业模板和话术库。
- 针对不同行业沉淀“岗位关键词 + 经历表达模板”。
- 让工具服务更宽的闲鱼人群。

#### 劣势 3：不解决简历视觉排版和多格式交付

它关注项目和面试材料，不是简历设计器。它虽然提到 VibeResume 做最后一公里，但自身不主打模板、PDF、DOCX、实时预览。

我们的机会：

- 将它的“JD→项目→面试”思路接入我们的“简历编辑器 + 模板 + 导出”。
- 做成从内容到交付文件的完整闭环。

### 2.4 我们应该吸收什么

| 可吸收能力 | 在我们项目中的落地方式 | 优先级 |
|---|---|---|
| JD 分析 | AI 工具箱增加“岗位 JD 解析” | P0 |
| 简历匹配评分 | 编辑器右侧显示 JD 匹配分和缺失项 | P0 |
| 面试追问 | 导出后生成“面试官可能追问清单” | P1 |
| 项目深挖 | 针对项目经历生成 STAR、技术难点、业务价值 | P1 |
| Readiness Gate | 导出前检查“能投、能面、能讲” | P1 |
| 结构化评分 | 所有评分维度可解释、可复核 | P0 |

## 3. 竞品二：ResumeCollection

### 3.1 项目概况

ResumeCollection 是一个简历模板收集仓库，项目描述是“求职/升学/程序员找工作简历模板大全”，README 中写明“收集全网的中英文简历，总有一款适合你”，并按中文简历、英文简历、多页简历、简历封面、简单表格简历分类。

GitHub 页面显示中文简历目录列出了大量模板项，至少包括 001 到 150 的模板预览和下载入口，覆盖产品经理、护士、财务、运营、教师、医护、设计师、软件开发、银行、市场等岗位。

参考来源：[ResumeCollection README](https://github.com/mmmlllnnn/ResumeCollection)

### 3.2 优势

#### 优势 1：模板数量多

这是它最直接的优势。大量模板能满足用户的第一直觉：

```text
我想先挑一个好看的样式
```

这类模板集合对闲鱼业务也有价值，因为客户往往会被“模板预览图”吸引。

#### 优势 2：行业覆盖广

从 README 列出的模板名称看，它覆盖了：

- 产品经理。
- 软件开发。
- 新媒体运营。
- 市场营销。
- 财务会计。
- 护士医护。
- 教师。
- 设计师。
- 银行金融。
- 行政人事。
- 应届生。

这提醒我们：模板中心不能只做“互联网科技风”。闲鱼客户很多并不是程序员。

#### 优势 3：Word 模板天然可编辑

它的模板多是 Word 文档，对用户来说优势明显：

- 下载后可直接编辑。
- 不依赖在线工具。
- 客户熟悉 Word。
- 适合最终交付 DOCX。

#### 优势 4：预览 + 下载的展示方式简单有效

README 中模板以“简历预览 / 下载”的方式排列。虽然不算高级，但对模板类产品非常直接。

我们模板中心也应该避免过度抽象，至少要做到：

- 缩略图足够大。
- 分类清晰。
- 一键套用。
- 一键导出预览 PDF。

### 3.3 劣势

#### 劣势 1：版权和授权风险不明确

README 说明模板“收集自网络，免费分享”。这对个人学习可能没问题，但如果我们用于商业接单，存在版权和授权风险。

我们不能直接把这些模板拿来商用。可以借鉴风格，但必须：

- 自制模板。
- 使用明确可商用素材。
- 或只作为设计灵感，不直接复制文件。

#### 劣势 2：只是模板集合，不是生产工具

它没有解决：

- 简历内容怎么写。
- 如何匹配 JD。
- 如何 AI 优化。
- 如何批量处理客户订单。
- 如何导出多版本。
- 如何管理客户项目。

我们的机会是把“模板资源”变成“模板 + 内容 + AI + 交付流程”。

#### 劣势 3：模板质量不稳定

模板集合通常会有这些问题：

- 有些模板过度设计，不 ATS 友好。
- 有些模板行业过时。
- 有些模板排版不适合一页。
- 有些模板字体、图标、颜色不统一。
- Word 兼容性可能不稳定。

README 里也提到如出现兼容性问题，需要自行修改 `docx` / `doc` 后缀。这说明模板交付层可能存在格式稳定性问题。

#### 劣势 4：缺少筛选和推荐逻辑

模板多本身会带来选择困难。用户不知道：

- 应届生选哪个。
- 产品经理选哪个。
- 外企英文简历选哪个。
- 哪个更 ATS 友好。
- 哪个适合一页。

我们的模板中心要比它更进一步：不是展示 150 个模板，而是让用户快速选到合适的 3 个。

### 3.4 我们应该吸收什么

| 可吸收能力 | 在我们项目中的落地方式 | 优先级 |
|---|---|---|
| 大量模板预览 | 模板中心做缩略图网格 | P0 |
| 行业标签 | 模板增加岗位/行业/风格/页数/ATS 标签 | P0 |
| Word 交付 | 第二阶段重点完善 DOCX 导出 | P1 |
| 模板分类 | 互联网、金融、医护、运营、设计、应届生、英文 | P0 |
| 预览下载体验 | 模板卡片支持预览、套用、导出示例 | P1 |

## 4. 竞品三：resume-design / 猫步简历

### 4.1 项目概况

猫步简历是一个开源免费的简历设计制作网站。README 中说明其技术栈为 Vue3 + TypeScript + Vite + Element Plus + Pinia，后端技术栈为 Nest.js，数据库 MongoDB，文件存储 MinIO。

README 中列出的能力包括：

- 用户邮箱验证码登录注册。
- 个人中心。
- 在线制作设计器。
- 积木创作设计器。
- 导出超高清 PDF。
- 支持在线和导出 JSON 数据。
- 评论。
- Word 简历模板下载。
- PPT 模板下载。
- 保存草稿。
- 积分简币体系。
- 主题任意更改，包括字体、颜色、背景、间距。
- 后台管理系统。
- 网站数据统计。

参考来源：[resume-design README](https://github.com/Hacker233/resume-design)

### 4.2 优势

#### 优势 1：完整产品化程度高

它不是简单 Demo，而是一个完整在线产品。功能覆盖：

```text
用户系统 → 模板 → 设计器 → 导出 → 草稿 → 后台 → 数据统计
```

这说明它适合做公网平台，而不是单人本地工具。

#### 优势 2：设计器能力强

README 中提到两款设计器：

1. 在线制作设计器：专注简历数据，提供丰富模板、预置模块、模块拖动排序。
2. 积木制作设计器：更灵活，可以制作简历、海报、Logo 等，组件粒度更小。

这给我们的启发：

- MVP 不做积木式自由设计器。
- 但要做简历模块化编辑器。
- 工作经历、项目经历、教育背景、技能等要能拖拽排序。
- 模板主题要支持颜色、字体、间距的轻量调整。

#### 优势 3：主题定制能力强

它支持字体、颜色、背景、间距等主题任意更改。这比固定模板更灵活。

我们的模板系统也应该做“模板 + 主题变量”，而不是每个颜色都做一套模板。

建议模板变量：

```json
{
  "primary_color": "#2563eb",
  "font_family": "Source Han Sans",
  "font_size": 10.5,
  "page_margin": 15,
  "section_spacing": 12,
  "line_height": 1.55
}
```

#### 优势 4：导出格式丰富

GitHub 项目描述和 README 都提到 PDF、图片、JSON、Markdown、Word 模板下载等能力。尤其是 JSON 数据导出，对后续版本管理、模板切换、跨端复用非常关键。

我们的现有方案已经有 JSON 导入导出的方向，应保留并提前实现。

#### 优势 5：后台管理和数据统计完整

它有管理端、网站数据统计、后台配置。这对 SaaS 很重要。

我们当前不做公网平台，但可以借鉴一部分本地运营统计：

- 本月处理简历数。
- 平均处理时长。
- 常用模板。
- 常用 AI 动作。
- 导出次数。
- 已交付 / 待复核数量。

### 4.3 劣势

#### 劣势 1：对我们当前阶段过重

猫步简历功能很多，但我们前期只是验证闲鱼接单模式。如果照搬它，会陷入过度开发：

- 登录注册暂时不需要。
- 积分体系暂时不需要。
- 评论暂时不需要。
- 后台管理暂时不需要。
- 积木设计器暂时不需要。

我们的目标是先接单交付，而不是做完整 SaaS。

#### 劣势 2：自由设计器可能让用户偏离“简历质量”

积木式设计器虽然灵活，但对简历产品有副作用：

- 用户容易过度设计。
- ATS 友好性下降。
- 排版不可控。
- 导出稳定性变差。
- 接单交付时效率下降。

我们应该坚持“模板约束 + 少量可调变量”，而不是无限自由。

#### 劣势 3：公网产品涉及更多运维和合规成本

它的完整方案包含后端、数据库、文件存储、用户系统。这意味着：

- 部署复杂。
- 数据隐私压力大。
- 文件存储成本。
- 账号安全问题。
- 后台运营成本。

我们当前的优势正好相反：本地运行、低成本、隐私可控。

#### 劣势 4：不是面向接单交付的工作台

猫步简历更像用户自己制作简历的平台，而不是运营者帮客户批量处理的后台。

它不一定天然支持：

- 客户项目状态。
- 客户修改次数。
- 交付版本记录。
- 一键生成优化说明。
- 闲鱼接单 SOP。
- 交付前事实检查。

这些是我们的机会。

### 4.4 我们应该吸收什么

| 可吸收能力 | 在我们项目中的落地方式 | 优先级 |
|---|---|---|
| 模块化编辑器 | 简历板块可增删、排序、复制 | P0 |
| 模板主题变量 | 支持主色、字体、间距、页边距调整 | P1 |
| JSON 数据导出 | 作为备份和跨端迁移格式 | P0 |
| 多格式导出 | PDF、DOCX、TXT、JSON、Markdown | P1 |
| 模板开发体系 | 模板由 HTML/CSS + schema 描述 | P1 |
| 本地数据统计 | 简化版运营看板 | P2 |

## 5. 我们当前方案的不足

结合竞品看，我们现有方案已经有方向，但有几个明显短板。

### 5.1 模板体系还不够强

当前方案里 MVP 只规划 modern、classic、compact 三套模板。这对验证可以，但对闲鱼转化不够。

客户看到服务时，首先会问：

- 有没有我这个行业的模板？
- 有没有好看的模板？
- 能不能 Word 编辑？
- 有没有和别人不一样的样式？

因此我们需要从第一版就设计好模板标签和扩展机制。

### 5.2 AI 工作流还偏“简历内部优化”

当前 AI 功能包括诊断、优化、JD 匹配、翻译、语法检查，但还不够像 shushu 那样形成“求职闭环”。

我们应该补：

- JD 解析。
- 项目深挖。
- 面试追问。
- 投递检查表。
- 优化说明自动生成。

### 5.3 缺少模板质量标准

模板不能只看颜值。需要建立模板审核标准：

- 是否 ATS 友好。
- 是否适合一页。
- 是否适合中文。
- 是否适合 DOCX。
- 是否适合打印。
- 是否字体可商用。
- 是否手机端可预览。

### 5.4 缺少“接单交付”专属功能

竞品多面向用户自助，我们面向本地接单，应更重视：

- 客户项目状态。
- 修改次数。
- 交付版本。
- 优化说明。
- 隐私清理。
- 文件命名。
- 交付检查。

这部分是我们做出差异化的关键。

## 6. 我们要怎么做得比他们更好

### 6.1 产品定位上：做“简历交付工作台”，不是模板网站

竞品对比：

| 类型 | 竞品表现 | 我们的打法 |
|---|---|---|
| 模板集合 | ResumeCollection 强 | 我们不拼数量，拼精选和推荐 |
| 在线设计器 | 猫步简历强 | 我们不拼复杂设计器，拼交付效率 |
| AI 工作流 | shushu 强 | 我们吸收 JD→简历→面试闭环，并做成 UI |

我们的核心定位：

```text
给接单者使用的 AI 简历交付工作台
```

不是：

```text
给所有人自由设计简历的在线平台
```

### 6.2 模板上：少而精，带标签，能推荐

不要一开始做 100 个模板。我们应该先做 12 个高质量模板，每个模板都有明确标签。

第一批建议：

| 模板 | 适合人群 | 标签 |
|---|---|---|
| 互联网简约 | 产品、运营、技术 | ATS、单页、蓝色 |
| 技术工程师 | 前端、后端、测试、算法 | 项目突出、技能突出 |
| 应届生通用 | 校招、实习 | 教育背景、校园经历 |
| 金融咨询 | 金融、咨询、投行 | 稳重、黑白、正式 |
| 国企行政 | 行政、人事、文职 | 稳定、传统、简洁 |
| 医护护理 | 护士、医生、药学 | 证书突出、经历清晰 |
| 教师教育 | 教师、教培、辅导员 | 证书、荣誉、经历 |
| 设计创意 | UI、视觉、品牌 | 作品集、视觉化 |
| 市场销售 | 市场、销售、商务 | 业绩数字突出 |
| 英文简历 | 外企、留学、英文岗位 | 英文、ATS |
| 双栏优雅 | 通用 | 信息密度高 |
| 极简 ATS | 投递系统优先 | 纯文本友好 |

关键功能：

- 模板卡片显示“适合岗位”。
- 根据目标岗位推荐模板。
- 模板支持主色切换。
- 模板支持一页模式。
- 每个模板有 PDF 示例。

### 6.3 AI 上：从“润色”升级为“岗位匹配引擎”

竞品中的 shushu 给了很好的启发：用户真正需要的是匹配岗位、拿到面试。

我们 AI 工具箱应该改成 6 个核心按钮：

| AI 工具 | 作用 | 价值 |
|---|---|---|
| JD 解析 | 提取岗位关键词、硬性要求、隐性要求 | 优化方向更准 |
| 简历诊断 | 找出结构、表达、匹配问题 | 先诊断再修改 |
| 单条改写 | 生成稳妥版、强化版、精简版 | 降低 AI 编造风险 |
| 量化成果 | 提醒哪些内容可补数据 | 让经历更硬 |
| 面试追问 | 根据简历生成追问题 | 提升高客单价 |
| 优化说明 | 自动生成交付说明 | 提升专业感 |

AI 输出必须遵守：

- 不直接覆盖。
- 不编造。
- 待确认信息高亮。
- 每条建议有理由。
- 每次导出前强制事实检查。

### 6.4 编辑器上：做“模板约束下的高效编辑”

不要做猫步那种完整积木设计器。我们应该做：

```text
结构化表单 + 实时 A4 预览 + 模块排序 + 主题变量
```

必做能力：

- 工作经历、项目经历可拖拽排序。
- 每条 bullet 可单独 AI 改写。
- 模板一键切换，内容不丢。
- 主题颜色可换。
- 字号/行距/页边距可微调。
- 自动一页模式。
- 预览和导出一致。

不做或后做：

- 任意拖拽文本框。
- 海报式自由布局。
- 自定义复杂背景。
- 多人协作。

### 6.5 导出上：比模板集合更稳定，比在线平台更可控

我们要提供清晰的导出策略：

| 格式 | 优先级 | 用途 |
|---|---:|---|
| PDF | P0 | 标准交付 |
| DOCX | P1 | 客户可编辑 |
| JSON | P0 | 数据备份、后续小程序复用 |
| TXT | P1 | ATS / 纯文本 |
| Markdown | P2 | 程序员、GitHub 简历 |

导出前检查：

- 中文字体是否正常。
- 页面是否溢出。
- 是否超过一页。
- 联系方式是否完整。
- 是否包含待确认数据。
- 是否选择了合适模板。

### 6.6 接单流程上：做竞品没有的“交付管理”

这是我们的真正优势。

新增模块建议：

| 模块 | 功能 |
|---|---|
| 客户项目状态 | 草稿、待复核、待客户确认、已交付、已归档 |
| 修改次数 | 记录第几次修改，避免售后失控 |
| 交付版本 | v1、v2、最终版 |
| 优化说明 | 自动生成“改了什么、为什么改” |
| 文件管理 | 原始文件、导出文件、最终文件分开 |
| 隐私清理 | 已交付后清理上传文件 |
| 话术模板 | 收集资料、交付、催好评、拒绝造假 |

这部分比模板和设计器更贴合你的商业模式。

### 6.7 小程序化上：提前把核心能力放后端

后续要做微信小程序，所以现在要避免把业务逻辑写死在 React 里。

应该放后端：

- AI 调用。
- JD 解析。
- 简历评分。
- resume_data 校验。
- 模板渲染。
- PDF / DOCX 导出。
- 文件管理。
- 交付记录。

前端只做：

- 表单编辑。
- 预览。
- 交互状态。
- 配置入口。

未来小程序可复用：

- resume_data。
- API。
- AI Prompt。
- 模板渲染。
- 导出服务。

## 9. 对现有开发方案的修改建议

### 9.1 模板中心需要升级

原方案：

```text
MVP 3 套模板，后续扩展到 8-10 套
```

建议改成：

```text
MVP 3 套模板 + 完整模板标签体系
第二阶段 12 套精选行业模板
每套模板包含：缩略图、行业标签、岗位标签、ATS 标签、页数标签、导出测试状态
```

### 9.2 AI 工具箱需要从功能按钮升级为工作流

原方案：

```text
AI 优化、AI 生成、AI 润色、技能匹配
```

建议改成：

```text
JD 解析 → 简历诊断 → 板块优化 → 量化成果 → 面试追问 → 交付说明
```

### 9.3 增加“交付中心”

新增页面或模块：

```text
交付中心
- 当前客户信息
- 当前版本
- 修改次数
- 导出文件
- 优化说明
- 交付检查
- 隐私清理
```

这会让我们和普通简历生成器拉开差距。

### 9.4 模板不要直接复用网上收集文件

ResumeCollection 可以作为视觉参考，但不要直接使用其 Word 文件做商业交付。

建议：

- 参考其行业覆盖和命名方式。
- 自己用 HTML/CSS 重建模板。
- 确保字体和图标可商用。
- 每个模板建立导出测试样例。

### 9.5 不做猫步式积木设计器

短期明确不做：

- 任意组件拖拽。
- 物料市场。
- 模板投稿审核。
- 积分体系。
- 评论。

我们做：

- 简历模块排序。
- 主题变量。
- 模板切换。
- 自动一页。

这样更符合接单效率。

## 10. 推荐的新产品路线

### 阶段一：高保真前端原型

目标：让界面像一个可用产品。

做：

- 首页。
- 简历管理。
- 编辑器。
- 模板中心。
- AI 工具箱。
- 导入导出。
- 设置页。
- 本地假数据。

不做：

- 真实 AI。
- 真实导出。
- 后端。

### 阶段二：本地 MVP

目标：能真实处理一份简历。

做：

- FastAPI。
- SQLite。
- DeepSeek 调用。
- PDF 导出。
- JSON 存储。
- JD 解析。
- 单条改写。
- 交付检查。

### 阶段三：接单增强版

目标：能稳定处理客户订单。

做：

- DOCX 导出。
- 导入解析。

- 修改记录。
- 隐私清理。
- 12 套行业模板。
- 面试追问。

### 阶段四：小程序预备版

目标：核心能力 API 化。

做：

- 手机端预览和支付闭环设计。

## 11. 最终差异化一句话

竞品要么是“模板很多”，要么是“设计器很强”，要么是“AI 工作流很深”。我们的最佳打法是把三者拆开吸收，再组合成一个更适合真实赚钱场景的产品：

```text
精选行业模板 + 可控 AI 优化 + JD 匹配评分 + 面试追问 + 本地隐私 + 接单交付管理
```

这样我们不是做一个普通简历生成器，而是在做一个能帮你快速交付、提高客单价、未来还能迁移到微信小程序的 AI 简历业务工作台。
