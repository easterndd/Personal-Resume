# ADR-0001: Modular Monolith and API-First Boundary

- 状态：Accepted
- 日期：2026-06-08
- 决策者：项目维护者

## Context

项目当前需要快速验证编辑、AI、导入导出和模板质量，同时未来要上线服务器并增加微信小程序。过早拆分微服务会增加部署、鉴权、追踪、数据一致性和本地开发成本。

## Decision

- 后端保持一个 FastAPI 模块化单体。
- Web 与未来小程序都通过稳定 REST API 使用核心能力。
- 模块按简历、AI、导入、导出、模板、设置划分。
- 数据库初期单库，文件初期本地存储。
- 仅当独立扩缩容、故障隔离或团队所有权形成真实需求时拆服务。

## Alternatives

### Pure Frontend

优点是部署简单，缺点是密钥、文件解析、PDF 导出和小程序复用困难，因此拒绝。

### Microservices

优点是独立部署，缺点是当前规模没有足够收益，因此暂不采用。

### Desktop Framework

Electron/Tauri 可以增强本地体验，但会增加打包和更新成本，且不利于后续 Web/小程序统一，因此不作为主路线。

## Consequences

正面：

- 本地开发和部署简单。
- 核心能力可被多个客户端复用。
- 后续可渐进替换数据库和文件存储。

负面：

- 必须主动维护模块边界。
- 单进程同步导出可能成为性能瓶颈。
- 前后端契约需要机器化管理。

## Verification

- 页面不直接依赖数据库和文件系统。
- 新小程序能力无需复制后端业务规则。
- 核心流程可通过 API 测试。
- 模块间不存在循环依赖。
